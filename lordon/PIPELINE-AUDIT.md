**Severity:** 🔴 Critical


**BUG 3 — Inventory Data Race Condition / Overwrite**  
**Lines:** 128–155 (entire inventory block)  
**Severity:** 🔴 Critical (causes stale/incorrect inventory in all downstream scripts)

```javascript
// CURRENT CODE — dashboard-data.js inventory block (lines 128–155):
const dashboard = {
  // ...
  inventory: {
    totalValue: totalInventoryValue,
    totalProducts: lsProducts.length,
    totalShopifyProducts: shopProducts.length,
    brandBreakdown: Object.entries(brandProducts)
      .map(([name, d]) => ({
        name,
        count: d.count,  // ← This is PRODUCT COUNT, not UNIT COUNT!
        cost: d.cost,    // ← This is supply_price per product, not cost of units in stock!
        retail: d.retail,
        margin: ...
      }))
    // ...
    // MISSING: totalUnits, totalRetail, byLocation, lastFullSync
  }
};
```

**Root Cause:** `build-lordon-dashboard-data.js` loops through `lsProducts` (the products API response) and aggregates `supply_price` and `price_including_tax`. This gives you the **catalog** cost/retail, not the **inventory** cost/retail. It has NO information about how many units are actually in stock.

`build-lordon-inventory.js` does this correctly — it fetches `/inventory?limit=1000` which has `current_amount` (actual units) and `average_cost` (actual cost basis per outlet).

**The race condition:** Whichever script runs LAST overwrites `dashboard.json → inventory`. The current state of `dashboard.json` was written by `build-lordon-dashboard-data.js` AFTER `build-lordon-inventory.js` ran — so the correct inventory data is now gone.

**Evidence:**
- `dashboard.json → inventory.brandBreakdown → ICHI → count: 209` (product records, wrong)
- `sell-through.json → byBrand → ICHI → currentStock: 3171` (actual units — from a PRIOR correct run)
- `dashboard.json → inventory` has NO `byLocation` key (inventory.js generates it, dashboard-data.js doesn't)
- `dashboard.json → inventory` has `totalShopifyProducts` (dashboard-data.js unique field) but no `totalUnits` (inventory.js unique field)

**FIX REQUIRED:**

Option A (recommended): Remove inventory computation entirely from `build-lordon-dashboard-data.js`. Let `build-lordon-inventory.js` own the inventory section exclusively. Merge inventory.js output into dashboard.json without overwriting:

```javascript
// In build-lordon-dashboard-data.js, REPLACE the inventory block with:

// Read existing inventory if present, don't overwrite it
const existingDashboard = fs.existsSync(path.join(DATA_DIR, 'dashboard.json'))

  : {};

  lastUpdated: new Date().toISOString(),
  briefing: { ... },
  analytics: { ... },
  // Preserve inventory from inventory.js run — don't overwrite!

  // ...

```

Option B: Run scripts in a guaranteed order with a master orchestrator script.

**BUG 4 — Shopify Customers: Only 50 Records (No Pagination)**  
**Lines:** 43–44  
**Severity:** 🔴 Critical (6,900 customers exist, only 50 are loaded)

```javascript
// CURRENT CODE (line 43):
customers: shopify('customers.json?limit=250'),
```

**Root Cause:** The Shopify customers fetch uses a single `limit=250` call without pagination. Shopify returns max 250 per page; the store has 6,900 customers. Only the first page (~50 records based on actual data) is loaded. The `total` count comes from `custCount.count` (the `/count` endpoint) which correctly shows 6,900, but only 87 of the 50-record sample have any orders.

**Confirmed by data:** `shopify-customers.json` has exactly 50 records. 43 of 50 have `orders_count: 0` and `total_spent: 0.00`.

**FIX REQUIRED:**
```javascript
// Replace single fetch with paginated fetch (same pattern as build-lordon-sell-through.js):
async function fetchAllShopifyCustomers() {
  const allCustomers = [];
  let pageUrl = `https://${SHOPIFY_STORE}/admin/api/2024-01/customers.json?limit=250&order=updated_at+desc`;
  while (pageUrl) {
    const { body, headers } = await fetchWithHeaders(pageUrl, { 'X-Shopify-Access-Token': SHOPIFY_TOKEN });
    allCustomers.push(...(body.customers || []));
    pageUrl = parseNextLink(headers.link);
    if (pageUrl) await sleep(400);
  }
  return allCustomers;
}
```

**Note:** 6,900 customers at 250/page = ~28 API calls. At Shopify's rate limit this takes ~15 seconds. Acceptable for a nightly build.

```
WATCH:         sellThrough >= 25% but below REPEAT_WORTHY threshold
```

**Discrepancies:**
1. `REORDER_NOW` trigger: script uses `sellThrough >= 70%` but plan says `100%` (sold out). The `remaining === 0` case is correct but the 70% threshold is wrong — a 70% sell-through with remaining stock is NOT REORDER_NOW.
2. `REPEAT_WORTHY` trigger: script uses `sellThrough >= 50%` but plan uses time-based thresholds (`>= 60% in ≤4 weeks` OR `>= 40% in ≤3 weeks`).
3. No `weeks` parameter passed to `calcAlert` — time-based thresholds can't be implemented.

**FIX REQUIRED:**
```javascript
function calcAlert(sold, remaining, sellThrough, wos, weeks) {
  if (sold === 0) return 'NEW';
  // Sold out with proven velocity = REORDER_NOW
  if (remaining === 0 && sold > 0) return 'REORDER_NOW';
  // High WOS urgency (< 3 weeks) AND high sell-through = also REORDER_NOW
  if (wos <= 3 && sellThrough >= 70) return 'REORDER_NOW';
  // Time-based REPEAT_WORTHY
  if ((weeks <= 4 && sellThrough >= 60) || (weeks <= 3 && sellThrough >= 40)) return 'REPEAT_WORTHY';
  // Broad WATCH
  if (sellThrough >= 25) return 'WATCH';
  if (sold > 0) return 'WATCH';  // has any sales but below 25%
  return 'OK';
}

// Update call to include weeks:
const alert = calcAlert(totalSold, totalRemaining, sellThrough, wos, weeks);
```
